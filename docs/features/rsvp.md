# RSVP System

- **Scope:** Party-based group RSVP, per-member submission, access control

## Overview

The RSVP system uses a party model: guests are grouped into parties (households/social groups). Each party receives a unique access code printed on their invitation. Entering the code on the login page grants access to RSVP for all party members.

---

## Party Model

```
Party "Demo Family" (code: DEMO-1234)
  ├── Jane Guest    (can_rsvp: yes, can_bring_plus_one: yes)
  └── John Guest    (can_rsvp: yes, can_bring_plus_one: no)
```

- Each party has a unique alphanumeric code (e.g. `SMITH-A1B2`)
- Codes are generated by the admin and printed on invitations
- A party session does NOT require a password — the code itself is the credential
- Each party member is a row in the `guests` table with a `party_id` foreign key

## RSVP Flow

### For Party Members (Party Code Login)

```
Guest visits /
  │
  ├─ Switches to "Party Code" tab
  │  Enters party code (e.g., DEMO-1234)
  │
  ▼
Lands on /rsvp
  Shows all party members, each with their own form
  │
  ├─ Jane Guest: [Name: Jane] [Attending? Yes/No] [+1: Sarah] [Submit]
  │
  └─ John Guest: [Name: John] [Attending? Yes/No] [Submit]
```

Each member's form can be submitted **independently** — you don't need to RSVP for everyone at once. Each submission upserts a single `rsvp_responses` row.

### For Individual Guests (Username/Password)

Guests who are not in a party have two possible states based on their user type and RSVP capability:

| `can_rsvp` | Behavior |
|---|---|
| `1` | Shows RSVP form for themselves only |
| `0` | Shows informational message: "no RSVP is required for you" |

### For Admin

Admin can access `/admin/rsvp` to view all submitted responses in a read-only table.

## Per-Member RSVP Form

Each form uses dynamic field names scoped to the member's ID:

```
<input type="hidden" name="member_id" value="2" />
<input name="name_2"    value="Jane Guest" />
<input name="attending_2" type="radio" value="yes" />
<input name="attending_2" type="radio" value="no" />
<input name="plus_one_2"  value="Sarah Guest" />   (only if can_bring_plus_one)
```

The server action `submitRsvp()` reads `member_id` from the form data, then validates:

| Session Type | Allowed Members |
|---|---|
| `admin` | Any member |
| `party` | Only members whose `party_id` matches the session's `partyId` |
| `guest` | Only the logged-in guest (their own `guestId`) |

## Submission

```typescript
// Server Action: submitRsvp()
// src/app/(main)/rsvp/actions.ts

export async function submitRsvp(prevState, formData) {
  // 1. Parse session
  // 2. Validate authorization (admin/party/guest check)
  // 3. Validate plus-one name if bring_plus_one=yes
  // 4. Upsert rsvp_responses row
  // 5. revalidatePath("/rsvp")
  // 6. Return { success: true } or { error: "..." }
}
```

The action validates inputs and returns `{ success, error }`. The client component manages its own radio state locally — no sync-from-action needed since the user's selections are already correct.

The RSVP repository function `submitResponse()` does an `INSERT OR REPLACE` to handle both new submissions and updates.

## Rate Limiting

RSVP submissions are rate-limited per party using an in-memory sliding-window limiter (key: `party:{partyId}`). This is the same rate-limiting infrastructure used for login and questions — see [authentication.md](authentication.md#rate-limiting) for how it works.

| Config Key Constant | Default | Range |
|---|---|---|
| `RSVP_RATE_LIMIT_MAX_KEY` | 10 | 1–1000 |
| `RSVP_RATE_LIMIT_WINDOW_KEY` | 60 | 1–1000 seconds |

When the limit is exceeded, the server returns `{ error: "...", action: "cooldown", cooldownUntil }`. The client creates an `RSVP_LIMIT_UNTIL_KEY` cookie from the timestamp and displays a countdown timer ("Please wait Xs..."). The form is disabled during the cooldown. Config is editable via the Rate Limiting section on `/admin/rsvp`.

Note: RSVP rate limiting does **not** trigger IP bans — only login rate limiting does. See [ip-banning.md](ip-banning.md) for details.

## View-Only Guests

Guests with `can_rsvp = 0` see this on `/rsvp`:

> You're on the guest list, but no RSVP is required for you.
> If you have any questions, please contact us directly.

This is useful for:
- Children who are automatically attending
- Vendors (photographer, caterer)
- Guests whose attendance is already confirmed

## Database Tables

### `rsvp_responses`

```sql
CREATE TABLE rsvp_responses (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  guest_id INTEGER NOT NULL UNIQUE REFERENCES guests(id) ON DELETE CASCADE,
  guest_name TEXT NOT NULL,
  attending INTEGER NOT NULL DEFAULT 0,
  plus_one_name TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Each guest can have at most one response (`guest_id` is `UNIQUE`). Re-submitting updates the existing row.
